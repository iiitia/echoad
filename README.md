# EchoAd — Real-Time Ad Bidding Simulator

> **Focus:** Real-time ML inference + streaming pipeline + interactive visualization

A full-stack real-time ad bidding pipeline built for the EchoAd technical assessment. Synthetic ad requests are generated every 2 seconds to simulate real-time traffic, scored using a Gradient Boosting model trained on engineered features, and streamed live to a React dashboard via WebSockets.

🔗 **Live Frontend:** [echoad.vercel.app](https://echoad-iiitias-projects.vercel.app/)  
🔗 **Backend API:** [echoad.onrender.com](https://echoad.onrender.com)  
🔗 **WebSocket:** `wss://echoad.onrender.com/ws`

---

## What I Built

### The Pipeline (How data flows)

```
Producer (every 2s)
    │
    ▼
asyncio.Queue  ──►  Consumer + ML Model (predict_ctr)
                         │
                         ▼
                   WebSocket /ws
                         │
                         ▼
                 React Dashboard (Vercel)
```

1. **Producer** generates a fake ad request JSON every 2 seconds and pushes it onto an `asyncio.Queue`
2. **Consumer** pulls from the queue, runs the ad through the trained Gradient Boosting model, gets a CTR score (0.0 → 1.0), stores it in a ring buffer, and broadcasts it to all connected WebSocket clients
3. **Frontend** receives the scored ad in real time and displays it on the live dashboard

> **Kafka → asyncio.Queue pivot:** The brief called for Apache Kafka. After hitting Docker/port issues past the 20-minute threshold, I pivoted to Python's built-in `asyncio.Queue` (Fallback B). The producer/consumer contract is identical — only the queue transport layer changed.

## ML Model

- **Dataset:** Synthetic `ad_logs.csv` generated by `generate_data.py` with features: `age`, `device_type`, `site_category`, `time_of_day`, and binary target `click`
- **Model:** Gradient Boosting Classifier trained on engineered features (age, device, category, interactions)
- **Output:** CTR probability score between `0.0` and `1.0`
- **Serialisation:** `joblib` → `model.pkl`, loaded once at startup by `predict_ctr()`

---

## Feature Engineering

The model uses several engineered features to improve CTR prediction:

- **Age-based features:** `age`, `age_bucket`, `age_sq`
- **Behavioral flags:** `is_young`, `is_prime`, `is_senior`
- **Device signals:** `is_mobile`
- **Category risk scoring** based on domain knowledge
- **Interaction features:**
  - `category × device`
  - `mobile × age group`
  - `age × mobile`

These features help capture non-linear relationships and user behavior patterns that a raw feature set would miss.

---

## Model Improvements

- Switched from baseline models to **Gradient Boosting** for better predictive performance
- Handled class imbalance using **weighted training**
- Used **5-fold cross-validation** for reliable out-of-sample evaluation
- Tuned **decision threshold** (instead of default 0.5) for better F1 score on imbalanced CTR data

---

## Fallback Scoring (Heuristic Mode)

In case the ML model is unavailable (e.g., deployment constraints), a lightweight rule-based scoring function is used based on category, age, and randomness. This ensures the system remains fully functional even without ML dependencies — a deliberate design choice for production resilience.

## Backend — Key Design Decisions

### WebSocket with Ping/Pong Keepalive
Render's infrastructure closes idle WebSocket connections after 30 seconds. The backend responds to every `ping` message from the frontend with a `pong`, keeping the connection alive indefinitely.

```python
if text == "ping":
    await websocket.send_text("pong")
```

### Dead Client Cleanup
The consumer's `broadcast()` function collects failed clients in a separate `dead` set and removes them after iterating — never mutating a set mid-loop.

```python
async def broadcast(clients, data):
    dead = set()
    for client in clients:
        try:
            await client.send_json(data)
        except Exception:
            dead.add(client)
    clients -= dead
```

### Ring Buffer
Last 50 processed ads are kept in `app.state.recent_ads` for the `/ads` polling endpoint, so clients that missed messages can catch up.

## Frontend — Key Design Decisions

### useWebSocket Hook — Exponential Backoff
The custom hook auto-reconnects on any unexpected disconnect using exponential backoff (1s → 2s → 4s → … capped at 30s). Render's free tier spins down after inactivity, so this is essential.

### 25s Keepalive Ping
Frontend sends `"ping"` every 25 seconds — just under Render's 30s idle timeout — to hold the connection open while the dashboard is active.
